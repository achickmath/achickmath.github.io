Technical Projects

OmniBlade - Adaptive Motion Controller

Discovery Project | March 2026 | Tech(Hack) Hackathon

Overview

OmniBlade is an adaptive game controller that uses arm motion instead of traditional buttons and joysticks. Built for players with limb differences, tremors, or limited hand mobility, it turns natural arm gestures into game inputs. Tilt your arm to move, flick your wrist to attack. No buttons, no setup, just motion.

The Problem

Accessibility controllers exist, but they all assume you can use your fingers. The Xbox Adaptive Controller remaps buttons to bigger buttons. Eye trackers need cameras and calibration. Switch access devices require mounting hardware and more setup time than actual gameplay. Every solution adds complexity while still requiring fine motor control in the hands.

For players with limb differences or limited hand mobility, the problem was never which button to press — it's that buttons were never designed for them in the first place. OmniBlade eliminates buttons entirely. If you have an arm, you can play.

Technical Implementation

The hardware is an ESP32 DEVKIT V1 paired with an MPU-6050 6-axis IMU (accelerometer + gyroscope) mounted in a custom 3D-printed forearm brace. The ESP32 reads sensor data over I2C at 50Hz and runs a complementary filter to fuse accelerometer and gyroscope readings into stable roll and pitch angles.

The complementary filter balances both sensors — trusting the gyroscope for fast motion and the accelerometer for slow corrections. This prevents the drift you'd get from using just the gyro and the noise you'd get from just the accelerometer. Flick gestures are detected by monitoring gyroscope spike thresholds with cooldown logic to prevent false triggers.

Data streams wirelessly over Bluetooth Serial to a Python script on the host PC. The script maps tilt angles to WASD keypresses and flick events to virtual Xbox controller buttons and mouse clicks. This gives simultaneous keyboard, mouse, and gamepad output, making the controller compatible with any game.

Results

We demonstrated OmniBlade live with Dark Souls III, Hades, and Forza Horizon 5 — three completely different control schemes running off the same hardware. The system worked without calibration or remapping between games.

The biggest technical challenge was tuning the complementary filter's alpha value and flick detection thresholds to feel natural. Too sensitive and every small movement triggered inputs. Too conservative and gestures didn't register. Getting stable orientation from noisy IMU data required extensive iteration.

What I Learned

This project taught me I2C communication and sensor fusion on embedded hardware, complementary filtering for IMU orientation estimation, Bluetooth Serial implementation on ESP32, and virtual input simulation on Windows via Python. I also learned how to design gesture vocabularies around physical constraints — mapping complex game mechanics to only four flick directions forced hard decisions about which inputs actually matter.

Future Development

The current build is just the foundation. Next steps include replacing the complementary filter with a Madgwick filter for better accuracy under fast motion, adding a second IMU on the upper arm to detect relative motion between arm segments, and training a gesture classifier that goes beyond four flick directions. The end goal is a controller expressive enough to handle any game, for any player.

Tech Stack: ESP32, MPU-6050 IMU, Python, Bluetooth Serial, Complementary Filter, Virtual Input Simulation
